# 📘 PRD Balans 2.0 (ChatGPT)

# **Product Requirements Document**

**Версия:** 2.0

**Дата:** февраль 2025

**Статус:** рабочая версия

**Формат:** Структура стадий → блоков → полей (“одна дорожка”)

**Технологии:** Next.js 16 (SSR, Server Actions, Layouts, Loading), Supabase, Tailwind, ShadCN UI

---

# **1. Краткое описание**

**Balans** — это мобильное-ориентированное PWA-приложение для управления дизайн-проектами интерьера.

Главная цель — **дать дизайнеру единую линейную структуру работы над проектом**, включающую:

- сбор ТЗ
- планировку
- коллажи и подбор материалов
- визуализации
- рабочую документацию
- спецификации
- авторский надзор
- согласования с клиентом
- библиотеку материалов
- управление файлами
- историю действий

Приложение открывается как **одна дорожка**, в которой проект ведётся последовательно — от первой стадии до последней.

Каждая стадия имеет блоки, блоки — поля/формы/галереи.

---

# **2. Целевая аудитория**

**Основные пользователи:**

1. **Дизайнеры интерьера** — фрилансеры и небольшие студии (1–5 человек).
2. **Руководители студий** — управление несколькими проектами и дизайнерами.
3. **Младшие дизайнеры** — выполнение задач и наполнение полей.
4. **Клиенты** — просмотр и утверждение решений.

---

# **3. Видение продукта**

**Balans = линейный процесс → меньше хаоса → больше контроля.**

Приложение должно:

- убрать хаос из Telegram/Notion/Google Drive/Whatsapp/Excel
- позволить вести ВСЮ работу над проектом в одном месте
- работать идеально на телефоне
- экономить 20–40% времени дизайнера
- автоматически генерировать документы (ТЗ, спецификации, отчёты авторского надзора)

---

# **4. Принципы UX**

1. **Единая дорожка (core concept)**
    
    Проект — это последовательность стадий.
    
    Открываешь проект → видишь linear flow → открываешь стадию.
    
    Никакого “меню”, “древовидных структур”, “бесконечных вкладок”.
    
2. **Фокус на мобильную версию (приоритет E)**
    - идеальная работа на телефоне
    - свайпы
    - большие элементы UI
    - модальные формы
3. **Простая структура данных**
    - стадия → блоки → поля
4. **Минимум полей, максимум автоматизации**
    
    Адрес, площади, стоимость → автоподсказки.
    
    Материалы → подгружаются из библиотеки.
    
5. **Единый визуальный стиль**
    
    Чёрно-белая минималистичная база + бежевые акценты (стиль BEZ.LISHNEGO).
    
6. **Принцип «Единых сущностей»**
    
    Balans использует систему сквозных сущностей:
    
    - материалы
    - контакты (клиенты и подрядчики)
    
    Эти сущности живут вне конкретного проекта, могут переиспользоваться, а проект лишь **ссылается** на них.

# **5. Структура проекта (основа всего)**

Проект состоит из фиксированного набора стадий.

Студия может скрывать/добавлять необязательные стадии.

## **Стадии Balans:**

### **1. Техническое задание**

Блоки:

- Основная информация
- Информация о жильцах
- Список помещений
- Демонтаж
- Монтаж
- Инженерия (вентиляция, вода, отопление, кондиционирование)
- Мебель и оборудование

Стадия завершается после:

- заполнения всех блоков
- утверждения клиентом

---

### **2. Планировочные решения**

Блоки:

- Обмерный план
- Фотообмеры
- Анализ
- Варианты планировок
- Финальный план
- Экспликация

Согласование финального варианта клиентом — обязательное.

---

### **3. Коллажи и подбор материалов**

Блоки:

- Мудборды
- Коллажи по помещениям
- Материалы (из библиотеки)
- Мебель
- Освещение
- Сантехника
- Техника

При добавлении нового материала — система спрашивает:

**“Добавить в библиотеку?”**

---

### **4. Визуализация**

Блоки:

- Исходники
- Черновые рендеры
- Комментарии
- Финальные рендеры
- Согласование

---

### **5. Рабочая документация**

Блоки:

- Набор чертежей
- Загруженные PDF
- Комплект документации

---

### **6. Спецификации (таблицы)**

Блоки:

- Мебель
- Освещение
- Материалы
- Декор
- Сантехника
- Техника

Позиции берутся:

- из библиотеки
- создаются вручную
    
    (с опцией “сохранить в библиотеку”)
    

---

### **7. Авторский надзор**

Блоки:

- Визиты
- Чек-листы
- Фотоотчёты
- Замечания
- Контроль поставок

---

# **6. Клиентский доступ**

У клиента есть отдельный режим просмотра:

### **Клиент может:**

1. **Просматривать проект в упрощённом виде**
    - только ключевые блоки
    - без внутренних служебных форм
2. **Утверждать стадии**
3. **Утверждать позиции спецификаций**
4. **Прикреплять свои файлы** (договор, БТИ, техпаспорта)

Клиент не видит:

- внутренние заметки
- черновые материалы
- историю действий команды

# **7. Основные функции**

## **7.1 Управление проектами**

- создание проекта
- карточки проектов
- статусы: в работе / пауза / завершён
- поиск

---

## **7.2 Линейная структура (дорожка)** — core feature

Проект всегда отображается как:

```
[ Техническое задание ] → [ Планировка ] → [ Коллажи ] → [ Визуализация ] → [ РД ] → [ Спеки ] → [ Надзор ]

```

---

## **7.3 Блоки и поля**

Блоки — это:

- галерея
- форма
- таблица
- файл-менеджер

Поля — текст, число, селект, фото, ссылка, галерея, список.

---

## **7.4 Библиотека материалов**

Помимо выбора материалов в блоках, появляется полноценная страница библиотеки:

### **Функции:**

- поиск по названию, бренду, артикулу
- фильтрация по категориям (мебель / свет / материалы / сантехника / техника / декор)
- просмотр карточки материала
- редактирование существующих материалов
- создание нового материала
- массовое редактирование (позже)
- связь материала с проектами и спецификациями

### **Карточка материала содержит:**

- название
- категория
- бренд
- модель
- артикул
- цена (от / вариантные цены)
- поставщик
- контакт поставщика
- изображение
- ссылка на товар
- комментарий дизайнера
- список проектов, где используется

### **Связь с контактами:**

- материал может иметь **прикреплённого подрядчика/поставщика**
    
    (из раздела “Контакты”)
    

---

## **7.5 Спецификации (как у programa.design, но лучше)**

- таблица из блоков
- позиции можно драгать
- фильтры по помещениям
- экспорт в PDF
- экспорты для подрядчиков
- версии спецификаций

---

## **7.6 Авторский надзор**

Каждый визит:

- дата
- чек-лист (какие работы проверены)
- фото
- комментарии
- статус проблемы (открыта/закрыта)
- генерация отчёта (PDF)

---

## **7.7 История и уведомления**

- назначили задачу
- добавили материал
- клиент утвердил стадию
- новый визит в авторском надзоре

Push:

- Telegram
- Email

## **7.8 Контакты (новая страница)**

Контакты вынесены в отдельный раздел, как в programa.design, но структурированы проще.

### **Категории контактов:**

1. **Клиенты**
2. **Подрядчики / Поставщики**
3. **Прочие контакты**

### **Карточка “Организация / Поставщик”:**

- название организации
- тип: поставщик / подрядчик
- сайт
- телефон
- email
- описание
- логотип
- связанные материалы (если это поставщик)
- список контактных лиц

### **Карточка контактного лица:**

- имя
- должность
- телефон
- email
- привязка к организации

### **Функции контактов:**

- создавать / редактировать организации
- добавлять контактных лиц
- смотреть список контактов
- фильтровать по типу
- искать по имени/организации
- прикреплять контакт:
    - **клиента → к проекту**
    - **поставщика/подрядчика → к материалу**

- дорожка
- стадии → блоки → поля
- ТЗ
- планировка
- материалы + библиотека
- коллажи
- визуализации
- спецификации (без экспорта PDF)
- авторский надзор
- клиентское утверждение стадий
- мобильная версия
- страница библиотеки материалов (просмотр, поиск, создание, редактирование)

- версии спецификаций
- Telegram push
- авторская аналитика
- AI
